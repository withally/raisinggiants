# Project Research Summary

**Project:** Your Parenting Blueprint (Raising Giants)
**Domain:** Personalized quiz-to-premium-PDF digital product (parenting assessment)
**Researched:** 2026-02-23
**Confidence:** MEDIUM-HIGH

## Executive Summary

Your Parenting Blueprint is a quiz-to-premium-PDF digital product in the consumer psychology space. Users complete a 10-20 minute multi-step quiz about their own childhood and parenting instincts, receive a free archetype result, and pay $14 to unlock a personalized 15-20 page PDF report generated by AI against a fixed template. The product's genuine competitive differentiator is its focus on the upbringing-to-parenting transmission mechanism — how a parent's own childhood shapes their instincts — territory that established competitors (Attachment Project, 16Personalities, Gottman) do not occupy. This angle has strong emotional resonance and no direct competitors at the product-market intersection.

The recommended technical approach is a Next.js 15.5 App Router application on Vercel, with Supabase for data persistence, Stripe Checkout for payments, Moonshot Kimi 2.5 for AI-personalized content generation, @react-pdf/renderer for server-side PDF generation, and Resend for transactional email delivery. All quiz state is held in a Zustand store client-side and persisted to Supabase in a single write on quiz completion. PDF generation is triggered exclusively via the Stripe `checkout.session.completed` webhook — never from the client-side success URL. Sections are generated in parallel via `Promise.all()` to stay within Vercel function timeout limits.

The highest-risk areas are: (1) the archetype framework itself — Sophia's clinical work is a hard prerequisite that blocks quiz design, scoring logic, PDF template, and AI prompt development; (2) AI content personalization quality — the entire value proposition depends on PDFs that feel "written for this specific person," not generic horoscope text; and (3) PDF delivery reliability under production load. All three risks are manageable with disciplined process: define the archetype framework first, invest in prompt engineering as a first-class deliverable, and implement an async generation queue from day one rather than synchronous inline generation.

## Key Findings

### Recommended Stack

See full detail: `.planning/research/STACK.md`

The stack is deliberately minimal and Vercel-native to reduce operational complexity. The most non-obvious decision is PDF generation: @react-pdf/renderer (JSX-based PDF authoring, `renderToBuffer()` in a Route Handler) rather than Puppeteer — the latter has a 10-15 second cold start on Vercel and will hit timeout limits when combined with multiple AI API calls. There is an active bug with `renderToStream()` in Next.js 15 App Router; the mitigation (`renderToBuffer()` + `serverExternalPackages` config) is documented and reliable.

**Core technologies:**
- **Next.js 15.5**: Full-stack framework — App Router eliminates separate API layer; Server Actions for form handling; Route Handlers for webhook/PDF endpoints
- **Supabase**: PostgreSQL + anonymous auth + Storage — quiz sessions, orders, PDF files; RLS enforces data access; service-role key server-only
- **Stripe Checkout (redirect)**: Payment + PCI compliance — webhook `checkout.session.completed` is the only authoritative fulfillment trigger; never use the success URL as the trigger
- **Moonshot Kimi 2.5**: AI content generation via OpenAI SDK pointed at Moonshot base URL — 256K context; $0.60/1M input tokens; generate each PDF section independently with `Promise.all()`
- **@react-pdf/renderer 4.3.2**: Server-side PDF generation — JSX authoring; `renderToBuffer()` only; add to `serverExternalPackages`; custom fonts via `Font.register()`
- **Resend**: Transactional email + PDF delivery — deliver via download link (signed URL), not attachment; react-email templates for brand consistency
- **Zustand 5 + React Hook Form 7 + Zod 4 + nuqs 2.8**: Quiz state — Zustand accumulates answers across steps; nuqs tracks step in URL for Back button; RHF validates per step; single Supabase write on completion

**Critical version notes:**
- `@supabase/supabase-js` 2.80.x requires Node.js 20+ (dropped 18 in 2.79.0)
- Use `@supabase/ssr` not the deprecated `@supabase/auth-helpers-nextjs`
- Tailwind 4.x uses `@theme` CSS directive — no `tailwind.config.js`
- shadcn/ui updated for Tailwind v4 + React 19 in February 2025 milestone

### Expected Features

See full detail: `.planning/research/FEATURES.md`

**Key competitive observation:** No competitor addresses the upbringing-to-parenting transmission mechanism. The "how your childhood shapes your parenting instincts" angle is genuinely unoccupied territory with deep emotional resonance. This is the product's core IP and must be reflected in quiz questions, PDF content, and all marketing copy.

**Must have (table stakes) — launch blockers:**
- Card-style quiz (one question per screen) with progress bar and back navigation
- Free archetype result page showing name + 2-3 paragraph resonance description + PDF preview samples
- Stripe Checkout at $14 with single clear price (no bundles, no upsells at checkout)
- PDF with fixed template structure + AI-personalized paragraphs
- PDF delivery to email after purchase (download link, not attachment)
- Landing page with credibility signals + quiz entry CTA
- "Why we ask this" helper text on sensitive questions
- Mobile-responsive design throughout (expected traffic is mobile-dominant)

**Should have (competitive differentiators):**
- Archetype-based result (6-8 named types, not a score) — identity-based result drives sharing and retention
- AI-personalized paragraphs within fixed PDF template — unique vs. all competitors who generate identical text per type
- Cultural background personalization — no competitor does this; direct differentiator for non-Western parents
- KOL research attribution in PDF — named researchers (not vague "studies show") with specific insights mapped to archetype
- Partner dialogue prompts as a PDF section — bridges self-reflection to relationship utility; plants v2 Couple's Blueprint seed
- Foundational patterns + watchouts dual structure — positive framing AND shadow/risk creates clinical depth

**Defer to v2+:**
- Couple's Blueprint (combined partner report) — high complexity; requires both partners completing quiz + report merging
- User accounts / dashboard — adds auth complexity; stateless flow is sufficient for v1
- Affiliate / referral program — needs purchase volume to validate economics
- Quiz result sharing card — low priority for a deeply personal product

**Anti-features (explicitly NOT building):**
- Retake quiz functionality (encourages gaming; undermines credibility)
- Fully AI-generated PDF (no fixed template) — quality control impossible; hallucination risk
- Real-time cloud-based draft saving — localStorage is sufficient; server-side state management for anonymous users is complex overhead

### Architecture Approach

See full detail: `.planning/research/ARCHITECTURE.md`

The architecture is a single Next.js application. Quiz state lives in Zustand (client-only) persisted to localStorage, flushed to Supabase in one write on completion. Supabase anonymous sign-in (`signInAnonymously()`) creates a session anchor without requiring account creation — the resulting `user_id` is passed through Stripe session metadata to the webhook handler. Webhook handler executes: signature verification → idempotency check → fetch quiz answers from Supabase → parallel AI content generation → PDF render → Supabase Storage upload → Resend email with signed URL. All of this must complete within Vercel's function timeout (60s on Pro; Hobby's 10s is insufficient — use Pro from day one).

**Major components:**
1. **Quiz Shell** (Client Component + Zustand) — step navigation, answer accumulation, localStorage persistence, single write to `/api/quiz` on completion
2. **Free Result Page** (Server Component) — reads archetype from Supabase, shows teaser + PDF sample, triggers Stripe checkout CTA
3. **Stripe Webhook Handler** (`/api/webhooks/stripe`) — signature-verified, idempotent; the sole authoritative fulfillment trigger
4. **PDF Generation Pipeline** (`/api/pdf` Route Handler + `lib/ai/` + `components/pdf/`) — parallel AI calls per section, `@react-pdf/renderer` `renderToBuffer()`, Supabase Storage upload
5. **Download / Success Page** (Server Component) — reads order fulfillment status; shows download button if ready, "check your email" if generating
6. **Supabase DB** — `quiz_sessions` (JSONB answers, archetype_id), `orders` (stripe_session_id, fulfilled_at, pdf_path)

**Build order dictated by dependencies:** Schema → Quiz Engine → Result Page → Payment Flow → AI Generation → PDF Rendering + Delivery → Landing Page & Polish.

### Critical Pitfalls

See full detail: `.planning/research/PITFALLS.md`

1. **Archetype scoring produces implausible results** — Sophia's clinical framework is a hard prerequisite for every downstream component. Zero quiz, scoring, PDF, or AI code should be written until the archetype system (6-8 types, scoring dimensions, question weights) is fully defined and Sophia has validated 20-30 simulated answer sets. This is the highest-priority dependency in the entire project.

2. **AI content feels generic ("horoscope problem")** — The core value proposition is PDFs that feel written for a specific person. Without verbatim quiz-answer injection into prompts and conditional content blocks, output will be polished generic text. Prompt engineering is a first-class deliverable. Before launch, run a blind review: a reviewer who hasn't seen the quiz answers reads 5 PDFs and must identify which answers drove which paragraphs.

3. **PDF delivery failure at the moment of peak excitement** — Synchronous PDF generation inline in the Stripe webhook handler will timeout under production load. Use async job queue from day one. Deliver via signed download URL (not email attachment — attachment triggers spam filters). Test to Gmail, Outlook, and Apple Mail before launch. Set up dead-letter queue monitoring for failed jobs.

4. **Quiz drop-off before delivering value** — Gate depth, not identity. Show archetype name + resonance description + PDF preview for free. Never ask for payment before quiz completion, never ask for email before archetype reveal. Free result page bounce rate above 70% before the CTA is a failure signal. This must be designed alongside the quiz, not as an afterthought.

5. **Supabase RLS disabled exposing sensitive psychological data** — This product collects childhood trauma indicators, family dynamics, and cultural background. RLS must be enabled on all tables at creation, verified through the SDK (not the SQL editor which bypasses RLS as postgres superuser), before any real user data is written.

**Additional critical risks:**
- Vercel function timeout (10s Hobby) — use Pro (60s) from day one; Hobby is insufficient for AI + PDF generation
- Stripe webhook signature not verified — implement `stripe.webhooks.constructEvent()` before the first sale; unverified webhook = fraud vector
- Moonshot Kimi rate limits blocking generation under launch load — implement async queue with exponential backoff retry before launch; validate API tier limits with actual account

## Implications for Roadmap

Based on research, the feature dependency chain and architectural build order suggest a 7-phase structure. The dominant constraint is that the archetype clinical framework must precede all engineering work.

### Phase 0: Archetype Framework (Pre-Engineering Prerequisite)
**Rationale:** PITFALLS.md identifies archetype scoring as the hardest dependency in the product. Every downstream component (quiz questions, scoring logic, PDF sections, AI prompts, result page copy) depends on this framework being complete. Starting engineering without it causes rework across every phase.
**Delivers:** 6-8 named archetypes with scoring dimensions, question weight matrix, Sophia validation against 20-30 simulated answer sets
**Addresses:** FEATURES.md archetype-based result (core IP); PITFALLS.md Pitfall 7 (implausible scoring)
**Avoids:** Rebuilding quiz, scoring, and PDF template after clinical review catches problems
**Note:** This is content/clinical work, not engineering. It must be done first. This phase has no research flag — it depends on Sophia's domain expertise, not external research.

### Phase 1: Data Foundation
**Rationale:** ARCHITECTURE.md identifies schema as the unblocking dependency for all downstream phases. Supabase tables, RLS policies, Storage bucket, and anonymous auth configuration must exist before any quiz data can be persisted.
**Delivers:** Supabase schema (`quiz_sessions`, `orders`), RLS policies verified through SDK, Storage bucket with signed URL access, anonymous sign-in enabled
**Uses:** Supabase `@supabase/ssr` 0.8.x, service-role key for webhook handlers only
**Implements:** Database layer, security boundary between anon and service-role clients
**Avoids:** PITFALLS.md Pitfall 8 (Supabase RLS disabled) — RLS enabled at table creation, tested before any real data
**Research flag:** Standard patterns (Supabase official docs are authoritative); skip research-phase

### Phase 2: Quiz Engine
**Rationale:** Quiz is the product entry point and generates the answers that feed every downstream system. Must be built before result page (needs archetype score) and before payment (needs quiz completion to trigger checkout). FEATURES.md identifies card-style quiz as the highest-value, lowest-cost P1 feature.
**Delivers:** Question bank (from archetype framework output), scoring logic (`lib/quiz/scoring.ts`), Zustand quiz store with localStorage persistence, quiz UI (one question per screen, progress bar, back navigation, "why we ask this" tooltips), `/api/quiz` route persisting session to Supabase
**Uses:** Zustand 5, React Hook Form 7, Zod 4, nuqs 2.8, shadcn/ui card components
**Implements:** Quiz Shell client component, Zustand store, mobile-first responsive layout
**Avoids:** PITFALLS.md Pitfall 3 (quiz fatigue) — psychological chapter breaks, non-linear progress bar, question variety; Pitfall 7 (implausible scoring) — scoring documented externally for Sophia review
**Research flag:** Needs `research-phase` for quiz UX architecture — specific question progression, chapter break UX patterns, and mobile interaction patterns (tap targets, keyboard behavior) warrant deeper research before implementation

### Phase 3: Free Result Page
**Rationale:** The result page is the primary conversion mechanism — it must create genuine desire for the paid PDF before the payment CTA appears. PITFALLS.md identifies this as the most commercially significant UX decision. It depends on Phase 2 (archetype scoring output) and is a prerequisite for Phase 4 (payment entry point).
**Delivers:** Archetype result page showing name + 3-4 paragraph resonance description + specific strength + one watchout + PDF preview/sample pages + CTA to purchase; teaser of partner dialogue prompts
**Implements:** Server Component reading from Supabase, archetype content templates per type
**Avoids:** PITFALLS.md Pitfall 1 (quiz drop-off) — result page designed to feel like "bonus unlock," not a wall; Pitfall 5 (credibility theater) — named KOL references on the result page, Sophia's credentials visible
**Research flag:** Standard patterns for result page conversion; skip research-phase. Use conversion data (scroll depth, CTA click rate) post-launch to iterate

### Phase 4: Payment Flow
**Rationale:** Depends on Phase 3 (result page is the entry point for checkout). Must be built before PDF pipeline since webhook triggers generation. PITFALLS.md Pitfall 4 (delivery failure) and Pitfall 6 (context break at Stripe redirect) are addressable at implementation time with known mitigations.
**Delivers:** `/api/checkout` route creating Stripe session with quiz session metadata, Stripe Checkout configured with archetype name in product description, `/api/webhooks/stripe` handler (signature-verified, idempotent, async job dispatch), checkout success page (polls order status, shows download or "check email")
**Uses:** Stripe Node SDK 20.3.x; webhook `checkout.session.completed`; Stripe CLI for local testing
**Implements:** Payment pattern from ARCHITECTURE.md — webhook-triggered fulfillment, idempotency guard on `orders.fulfilled_at`, success page reads from Supabase not from Stripe redirect
**Avoids:** PITFALLS.md Pitfall 4 (webhook must be authoritative, success URL is UI only); Pitfall 6 (customize Stripe Checkout branding maximally; evaluate embedded Payment Element before implementation — switching post-launch is non-trivial)
**Research flag:** Decision point — Stripe Checkout redirect vs. embedded Payment Element should be evaluated before implementation, not after. Research flag for this specific decision.

### Phase 5: AI Content Generation
**Rationale:** Depends on Phase 2 (quiz answers schema) and Phase 4 (webhook calls this). Must be built as a distinct engineering + content design phase — PITFALLS.md identifies prompt engineering as a first-class deliverable, not an implementation detail. Parallel generation with `Promise.all()` is not optional for timeout compliance.
**Delivers:** Moonshot Kimi 2.5 client (`lib/ai/moonshot.ts`), one prompt file per PDF section (`lib/ai/prompts/`), parallel section generation with individual retry and structured JSON output per section, async job queue architecture, blind review QA gate (5 PDFs reviewed by someone who hasn't seen quiz answers)
**Uses:** `openai` npm package pointed at Moonshot base URL, `kimi-k2.5` model, `response_format: { type: 'json_object' }` for structured output
**Implements:** Pattern 4 from ARCHITECTURE.md — sectional AI generation, parallelized `Promise.all()`, per-section retry on failure
**Avoids:** PITFALLS.md Pitfall 2 (generic AI content) — verbatim quiz-answer injection, conditional content blocks, cultural background specificity, shadow/watchout challenge alongside positive framing; Pitfall 9 (Kimi rate limits) — queue with exponential backoff retry built before launch
**Research flag:** Needs `research-phase` for prompt engineering patterns — structured output schemas per section, specificity techniques, conditional content block design, and Kimi 2.5-specific context window optimization

### Phase 6: PDF Rendering + Delivery
**Rationale:** Final step in the fulfillment pipeline. Depends on Phase 5 (AI content input). @react-pdf/renderer with `renderToBuffer()` is the confirmed approach; Puppeteer is explicitly excluded. Each PDF section is a separate React component for independent iteration.
**Delivers:** `BlueprintDocument.tsx` root document, one React component per PDF section (Cover, Archetype, Origin Profile, Cultural Insights, Watchouts, Reflection Prompts, Conversation Starters), PDF StyleSheet with brand fonts via `Font.register()`, Supabase Storage upload + signed URL generation, Resend email delivery (download link, not attachment), dead-letter queue monitoring for failed jobs
**Uses:** `@react-pdf/renderer` 4.3.2 with `serverExternalPackages` config, Resend 6.9.x with react-email, Supabase Storage private bucket
**Implements:** Full fulfillment pipeline; PDF section components with brand design (therapy workbook meets Canva premium)
**Avoids:** PITFALLS.md Pitfall 4 (delivery failure) — async job queue, "check your email" fallback message, download link not attachment, DKIM/SPF/DMARC verified, tested to Gmail + Outlook + Apple Mail
**Research flag:** Needs `research-phase` specifically for @react-pdf/renderer design patterns — layout capabilities and limitations, font loading, image handling, multi-page document structure, and brand design implementation within react-pdf's layout system

### Phase 7: Landing Page + Polish
**Rationale:** Marketing layer built last — there must be something worth selling before optimizing acquisition. FEATURES.md identifies landing page with credibility signals as P1, but it cannot be meaningfully completed until the product exists to demonstrate.
**Delivers:** Landing page with Sophia's credentials prominently visible, named KOL credibility signals (20-30 researchers), quiz start CTA, FAQ section addressing common concerns before quiz start, mobile-responsive layout, performance optimization
**Implements:** (marketing) route group in Next.js, static/ISR rendering for performance
**Avoids:** PITFALLS.md Pitfall 5 (credibility theater) — every archetype shown on landing must name at least 2 researchers; specific study references, not vague "research shows"
**Research flag:** Standard patterns; skip research-phase. Content decisions (KOL selection, Sophia's bio framing) are the critical work here, not technical decisions.

### Phase Ordering Rationale

- **Phase 0 before all engineering:** Archetype framework is a hard dependency for quiz questions, scoring weights, PDF section design, and AI prompts. Starting engineering before this is completed guarantees rework across every phase.
- **Data foundation before quiz:** Schema must exist to receive the session write on quiz completion.
- **Quiz before result page:** Result page displays the archetype computed from quiz answers; cannot build the display without the scoring output.
- **Result page before payment:** The result page is the entry point for the payment CTA. Payment flow integration starts from this page.
- **Payment flow before AI/PDF:** The Stripe webhook is the trigger for PDF generation. The integration point must be established before the generation pipeline is built.
- **AI generation before PDF rendering:** PDF rendering requires the AI-generated content structure. Prompts define the data schema; components consume it.
- **Landing page last:** Marketing layer wraps a complete product. Building it first creates misalignment between the promise and the delivery.

### Research Flags

**Phases needing deeper research during planning:**

- **Phase 2 (Quiz Engine) — UX architecture:** Question progression patterns, psychological chapter break design, non-linear progress bar behavior, and mobile quiz interaction patterns (keyboard avoidance, tap target sizing) need validated patterns before implementation. Building wrong quiz UX is expensive to rebuild.
- **Phase 4 (Payment Flow) — Stripe Checkout vs. embedded Payment Element:** This decision must be made before implementation. Switching post-launch is non-trivial. Research the conversion tradeoffs and implementation complexity of embedded Payment Element (Stripe Elements + Payment Element) vs. Checkout redirect with maximum customization.
- **Phase 5 (AI Content Generation) — Prompt engineering:** Structured output schemas, verbatim answer injection techniques, conditional content blocks, cultural personalization depth, and Kimi 2.5-specific prompt optimization all need research before implementation. Generic prompt patterns produce generic output — the opposite of the value proposition.
- **Phase 6 (PDF Rendering) — @react-pdf/renderer design patterns:** The library's layout system (Flexbox, no CSS Grid), font registration, image handling, multi-page document structure, and the gap between what's possible in web CSS vs. react-pdf need to be understood before design work begins. Finding out limitations mid-implementation requires redesign.

**Phases with standard patterns (skip research-phase):**

- **Phase 1 (Data Foundation):** Supabase official docs are authoritative and comprehensive. RLS patterns, anonymous auth, Storage signed URLs, and `@supabase/ssr` usage are all well-documented.
- **Phase 3 (Free Result Page):** Server Component data fetching from Supabase is well-established. Content and copy decisions are the primary work, not technical decisions requiring research.
- **Phase 7 (Landing Page):** Standard Next.js marketing page patterns. Content and KOL selection are the critical work.

### Gaps to Address

- **Archetype framework definition (Phase 0):** No research can substitute for Sophia's clinical work here. The number of archetypes (6-8 is the recommendation), their scoring dimensions, and the question-weight matrix must come from domain expertise. This gap blocks all engineering planning until resolved.
- **Kimi 2.5 rate limit tier for production account:** Research identified trial limits (~3 RPM) and the existence of paid tiers, but exact paid tier limits require verification against the actual Moonshot account. Validate at `platform.moonshot.ai` before finalizing the AI generation architecture. This affects queue design decisions.
- **Vercel function timeout budget:** Total AI generation time (6-7 parallel Kimi calls) + `renderToBuffer()` must be benchmarked against actual Kimi API latency before launch. Estimated 15-35 seconds for parallel generation — within Pro tier 60s limit, but must be validated empirically. If consistently above 45s, consider extracting to a dedicated async worker (Inngest or Trigger.dev).
- **Stripe Checkout redirect vs. embedded Payment Element:** Research identified this as a conversion-significant decision but did not resolve which approach is optimal for this product. Requires a focused research spike before Phase 4 implementation.
- **Cultural content variants per archetype:** The cultural personalization differentiator is a content dependency, not just engineering. Each archetype needs culturally-variant content written for the relevant cultural backgrounds asked in the quiz. The extent of this content work is unresolved and affects Phase 5/6 scope.
