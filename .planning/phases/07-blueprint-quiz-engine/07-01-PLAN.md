    **Migration 1** (`20260313000000_add_product_type_to_quiz_sessions.sql`):
    Add `product_type` column to existing `quiz_sessions` table:
    - `TEXT NOT NULL DEFAULT 'mirror'` with CHECK constraint `IN ('mirror', 'blueprint')`
    - Use `ADD COLUMN IF NOT EXISTS` for idempotency

    **Migration 2** (`20260313000001_create_bp_quiz_sessions.sql`):
    Create `bp_quiz_sessions` table with columns:
    - `id` UUID PRIMARY KEY DEFAULT gen_random_uuid()
    - `email` TEXT NOT NULL
    - `parent_status` TEXT NOT NULL CHECK IN ('current-parent', 'expecting', 'planning')
    - `status` TEXT NOT NULL DEFAULT 'in_progress' CHECK IN ('in_progress', 'completed')
    - `answers` JSONB
    - `dimension_scores` JSONB
    - `archetype_id` TEXT
    - `cultural_background` TEXT
    - `started_at` TIMESTAMPTZ NOT NULL DEFAULT now()
    - `completed_at` TIMESTAMPTZ
    - `created_at` TIMESTAMPTZ NOT NULL DEFAULT now()
    - `updated_at` TIMESTAMPTZ NOT NULL DEFAULT now()

    Enable RLS. Create index on email. Create `set_updated_at_bp_quiz_sessions` trigger referencing existing `handle_updated_at()` function (DO NOT redefine it — it already exists from Phase 1 migration).

    RLS policies (Blueprint sessions are email-keyed, no anon auth):
    - INSERT: service_role only (`WITH CHECK (true)`)
    - SELECT: anon + authenticated can read any row (`USING (true)` — session UUID is the credential)
    - UPDATE: service_role only (`USING (true)`)

    No `user_id` FK — Blueprint uses pay-first model, no anon auth flow.

    **API Route** (`app/api/bp-quiz-session/route.ts`):
    Follow the exact pattern from `app/api/quiz-session/route.ts` but target `bp_quiz_sessions` table.

    POST handler:
    - Accept `{ email, parentStatus }` in request body
    - Validate email (required, regex check)
    - Validate parentStatus (required, must be one of 'current-parent', 'expecting', 'planning')
    - Use `createAdminClient()` from `@/lib/supabase/server`
    - Insert into `bp_quiz_sessions` with `email`, `parent_status: parentStatus`, `status: 'in_progress'`
    - Return `{ sessionId: data.id }`
    - NO welcome email (Blueprint is pay-first; user already has an account interaction from payment)

    PATCH handler:
    - Accept `{ sessionId, answers, dimensionScores, archetypeId, culturalBackground }`
    - Validate sessionId required
    - Update `bp_quiz_sessions` setting: `status: 'completed'`, `answers`, `dimension_scores: dimensionScores`, `archetype_id: archetypeId`, `cultural_background: culturalBackground`, `completed_at: new Date().toISOString()`
    - Return `{ success: true }`
  </action>
