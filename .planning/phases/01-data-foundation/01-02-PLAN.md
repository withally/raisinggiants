  <action>
1. Add a `verify-rls` script to `package.json`:
   ```json
   "verify-rls": "npx tsx scripts/verify-rls.ts"
   ```

2. Create `scripts/verify-rls.ts` based on the research's "Anonymous Sign-In Verification Script" section but with the following complete test sequence:

   **Setup:** Create a Supabase client using `NEXT_PUBLIC_SUPABASE_URL` and `NEXT_PUBLIC_SUPABASE_ANON_KEY` (anon key only — NOT service_role). Load env vars from `.env.local` using dotenv or read directly.

   **Test 1 — Anonymous sign-in works:**
   - Call `supabase.auth.signInAnonymously()`
   - Assert: returns a user object with a valid UUID
   - Print: `PASS: Anonymous sign-in succeeded. User A: {id}`

   **Test 2 — User A can INSERT a quiz_session:**
   - Insert into `quiz_sessions` with `user_id: userA.id`, `status: 'in_progress'`
   - Assert: insert succeeds, returns the row
   - Print: `PASS: User A inserted quiz_session: {id}`

   **Test 3 — User A can SELECT their own quiz_session:**
   - Select from `quiz_sessions` where `id = sessionA.id`
   - Assert: returns exactly 1 row
   - Print: `PASS: User A can read their own session`

   **Test 4 — User A can UPDATE their own quiz_session:**
   - Update `quiz_sessions` set `status = 'completed'`, `completed_at = new Date().toISOString()` where `id = sessionA.id`
   - Assert: update succeeds
   - Print: `PASS: User A can update their own session`

   **Test 5 — Cross-user read isolation (CRITICAL SECURITY TEST):**
   - Sign out User A: `supabase.auth.signOut()`
   - Sign in as new anonymous User B: `supabase.auth.signInAnonymously()`
   - Attempt to SELECT `quiz_sessions` where `id = sessionA.id`
   - Assert: returns EMPTY result (0 rows) — NOT an error, just no rows
   - If rows returned: `SECURITY FAIL: User B CAN read User A's session — RLS policy not working!` and exit(1)
   - Print: `PASS: User B cannot read User A's session (RLS enforced)`

   **Test 6 — Cross-user insert rejection:**
   - As User B, attempt to INSERT into `quiz_sessions` with `user_id: userA.id` (not User B's own ID)
   - Assert: insert FAILS (RLS WITH CHECK should reject)
   - Print: `PASS: User B cannot insert a session for User A`

   **Test 7 — Storage bucket exists and is private:**
   - Call `supabase.storage.listBuckets()`
   - Find bucket with `id === 'blueprints'`
   - Assert: bucket exists AND `public === false`
   - Print: `PASS: blueprints bucket exists and is private`

   **Test 8 — Orders table SELECT isolation:**
   - Using a service_role client (separate createClient with service_role key), insert a test order for User A's session
   - Switch back to anon client as User B, attempt to SELECT that order
   - Assert: returns EMPTY (User B cannot see User A's orders)
   - Print: `PASS: User B cannot read User A's orders`

   **Cleanup:** Use service_role client to delete the test order and test quiz_session rows created during verification.

   **Final output:**
   ```
   === All Phase 1 RLS checks PASSED ===
   ```
   Or exit(1) on any failure with clear error message.

3. Install `dotenv` as a dev dependency if not already present, OR read `.env.local` manually. The script must work with `npx tsx scripts/verify-rls.ts` assuming `.env.local` has the Supabase connection details from `supabase start` output.

CRITICAL per research Pitfall 1: Never verify RLS in the SQL editor. This script is the ONLY valid verification surface. Every assertion tests through the anon-key SDK client with a real anonymous session.
